# Hamma Developer Instructions & Coding Rules

This file outlines the codebase architecture, tech stack constraints, security boundaries, and coding standards for **Hamma**. All agents and developers modifying this codebase must adhere strictly to these rules.

---

## 1. Project Identity & Vision

**Hamma** is a secure, multi-platform, local-first Flutter application (supporting Linux, Windows, macOS, Android, and iOS) designed to manage remote servers directly via SSH from the device. It is augmented by local AI-driven command suggestions, safety assessments, and visual management tools (SFTP, Docker, Services, and Fleet Dashboard).

*   **Product One-Liner:** *"Manage your server without writing commands."*
*   **Core Approach:** Prefer high-level visual actions (buttons, logs, explorer panels) over raw shell input. Keep raw terminal access as a fallback.

## 2. Core Architectural & Security Constraints (Non-Negotiable)

### 2.1 Zero-Trust Local AI (Loopback Enforcement)
To prevent prompt or logs exfiltration, the AI service layer enforces a hard loopback restriction (`127.0.0.0/8`, `::1`, `localhost`) for all local inference providers (Ollama, LM Studio, Jan, or fllama/llama.cpp native FFI).
*   **Implementation Rule:** Every local AI request/client constructor must validate the target URL using [OllamaClient.isLoopbackEndpoint](file:///home/botuser/git/hamma/lib/core/ai/ollama_client.dart#L41).
*   **Safety Rule:** Never bypass or disable this loopback check for local providers.
*   **Cloud Opt-In:** Cloud provider APIs (OpenAI, Gemini, OpenRouter) are allowed only via explicit user opt-in in settings. When active, a persistent, non-dismissible warning banner (e.g., *"⚠️ Cloud mode — prompts leave your device"*) must be displayed on the screen.

### 2.2 Encrypted Vault Storage (Credentials at Rest)
No credentials (SSH private keys, passwords, passphrases) may touch the disk unencrypted.
*   **Key Derivation:** Derive encryption keys using Argon2id via `pointycastle` (64 MB memory cost, 3 iterations, 4 parallelism).
*   **Encryption Scheme:** Encrypt all credentials using AES-256-GCM via `pinenacl` with cryptographically random salts and nonces. Store KDF salts securely in OS-level keychains using `flutter_secure_storage`.
*   **Lifecycle Rule:** Keep plaintext keys in memory only for the duration of active SSH/SFTP handshakes and sessions. Zero out key buffers from memory immediately upon session close or when the app is locked/backgrounded.
*   **Vault Storage Reference:** See [vault_storage.dart](file:///home/botuser/git/hamma/lib/core/vault/vault_storage.dart).

### 2.3 Direct SSH Connection First
*   All SSH and SFTP connections must run directly from the client device using [ssh_service.dart](file:///home/botuser/git/hamma/lib/core/ssh/ssh_service.dart) and [sftp_service.dart](file:///home/botuser/git/hamma/lib/core/ssh/sftp_service.dart) (leveraging the `dartssh2` package).
*   Do not build or route connections through backend-managed SSH proxy layers. The client device holds direct TCP control.

### 2.4 Command Execution Safeguards & Risk Scoring
All AI-generated or quick-action commands must be explicitly visible, editable, and confirmable by the user before execution.
*   **Mandatory Scans:** Always run commands through [CommandRiskAssessor](file:///home/botuser/git/hamma/lib/core/ai/command_risk_assessor.dart#L37) to determine risk levels (`low`, `moderate`, `high`, `critical`).
*   **Dangerous Commands (Critical/High Risk):** Flag any destructive command or action involving recursive deletion/permissions (`rm -rf`, `chmod -R`, `chown -R`), partitioning (`dd if=`, `mkfs`), system-wide state changes (`reboot`, `shutdown`, `systemctl stop`), firewall/network alterations (`iptables`, `ufw`), database resets, or Docker prunes. Show clear, high-contrast warnings and require extra user confirmation.

### 2.5 Zero Telemetry & Privacy
*   Do not add telemetry, analytics, remote crash reporters (e.g., Firebase Analytics), or phone-home tracking. Hamma respects absolute user privacy.

## 3. Technology Stack

*   **UI Framework:** Flutter (Material 3)
*   **Language:** Dart
*   **SSH & SFTP:** `dartssh2` for protocols, `xterm` for terminal UI emulation, and `flutter_pty` for local terminals.
*   **Database:** Hive (encrypted via derived keys) for server lists, preferences, and keys.
*   **Biometrics:** `local_auth` for OS-level Face ID, Touch ID, or fingerprint authentication.
*   **On-Device AI Engine:** `fllama` (Dart FFI bindings for `llama.cpp` native dynamic libraries) to load localized GGUF models, combined with loopback-bound Ollama API calls.
*   **Voice Operations:** `speech_to_text` and `flutter_tts` configured for on-device voice processing only.
*   **Testing Suite:** `flutter_test`, `mocktail` for mocks, and `fake_async` for virtual time test loops.
*   **Package Definitions:** Pinned in [pubspec.yaml](file:///home/botuser/git/hamma/pubspec.yaml).

## 4. Directory & Code Structure

Follow the feature-based architectural pattern:
*   [lib/main.dart](file:///home/botuser/git/hamma/lib/main.dart): App bootstrap, service injection, and core Material 3 theme configuration.
*   [lib/core/](file:///home/botuser/git/hamma/lib/core/): Decoupled, non-UI services, models, and security logic.
    *   `ai/`: AI providers, risk scoring, native llama.cpp bindings.
    *   `ssh/`: dartssh2 wrappers for session management and command execution.
    *   `vault/`: Encrypted credential engine (Argon2id + AES-256-GCM).
    *   `storage/`: Saved profile storage, API key storage, and configurations.
*   [lib/features/](file:///home/botuser/git/hamma/lib/features/): Feature-specific UI screens and controllers.
    *   `terminal/`: Interactive SSH terminal screen.
    *   `sftp/`: Visual file explorer, manager, and editor.
    *   `ai_assistant/`: Chat, command palettes, and copilot sheet.
    *   `servers/`: Add, edit, test, and dashboard views.
    *   `security/`: Vault credentials lock UI, PIN entry, biometrics.
*   [test/](file:///home/botuser/git/hamma/test/): Flat structure containing all unit, widget, and integration tests.

## 5. UX & Development Conventions

### 5.1 Simple First
*   The UI must prioritize buttons over commands. Non-experts should be able to perform primary server tasks (e.g., restarting services, viewing logs) without terminal interaction.
*   Always explain what is happening, what commands will do, and any potential side effects.

### 5.2 Avoid "Fake States"
*   Never display state statuses (e.g., "Connected", "Healthy", "Successful") based on predictions or static configuration. State must be verified by an active protocol check (e.g., SSH probe).

### 5.3 Code Design & Simplicity
*   Prioritize reliability and simplicity. Avoid overengineering; write clean, focused, and highly maintainable components.

### 5.4 Documentation Integrity
*   Preserve all existing comments, docstrings, and documentation unless explicitly instructed otherwise by the user.
*   Always write detailed docstrings for new services, methods, or models.

### 5.5 Test Requirements
*   Every new service, feature block, or helper must be accompanied by comprehensive tests in [test/](file:///home/botuser/git/hamma/test/).
*   Use `fake_async` for simulating time/timeouts without sleep commands in unit tests.
*   Ensure that new code doesn't break the high coverage baseline (700+ passing tests).

### 5.6 UI Styling & Design Aesthetics
*   Adhere to Material 3 guidelines using HSL-tailored colors, cohesive dark modes, and dynamic animations.
*   Use standard fonts: **Inter** for UI copy, **JetBrains Mono** for logs, terminals, and code snippets.
*   Ensure full responsive support (adapting between mobile touch targets and desktop wide views).
*   Avoid browser/native defaults and plain placeholders. All layouts must feel premium, visual, and polished.

### 5.7 File and Symbol Link Policy
*   When documenting plans or discussing files/code, developers and agents **must** format all file references and code symbol definitions (classes, methods, structs) as clickable markdown links using the `file://` scheme (e.g., `[filename](file:///path/to/file)`).

## 7. Guidelines for Adding AI Providers

When introducing a new AI inference provider:
1.  Implement the appropriate abstraction in [ai_provider.dart](file:///home/botuser/git/hamma/lib/core/models/ai_provider.dart) and configure settings UI in [settings_screen.dart](file:///home/botuser/git/hamma/lib/features/settings/settings_screen.dart).
2.  If the provider runs locally, ensure it is validated via [OllamaClient.isLoopbackEndpoint](file:///home/botuser/git/hamma/lib/core/ai/ollama_client.dart#L41) at the connection/client level.
3.  Store API keys securely using [api_key_storage.dart](file:///home/botuser/git/hamma/lib/core/storage/api_key_storage.dart).
4.  Add comprehensive validation and unit tests within [test/](file:///home/botuser/git/hamma/test/) targeting the provider behavior and loopback enforcement.
