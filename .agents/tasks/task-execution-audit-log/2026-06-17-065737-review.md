# Execution audit log for AI-proposed commands

The change adds an audit trail that records every command the AI copilot proposes and the user approves for execution. An immutable `ExecutionAuditEntry` model captures intent, command, risk level, server target, timing, and output. An `ExecutionAuditService` persists entries to FlutterSecureStorage with a 1000-entry cap. A Brutalist-styled `AuditLogScreen` provides search and filter-chip browsing. The integration hook in `AiCopilotSheet._runCommand` logs both successful and failed executions via `unawaited` fire-and-forget calls.

**Watch for:** Silent audit failures swallowed by `unawaited` with no error handling (confirmed). Unbounded stdout/stderr stored in encrypted storage can bloat the secure store (likely). The audit screen is created but never wired into navigation (confirmed). New `ExecutionAuditService()` instantiated on every command execution re-reads and re-writes the entire JSON blob from secure storage (confirmed).

## High-level view

The service layer stores the entire audit log as a single JSON string under one FlutterSecureStorage key. Every write deserializes the full list, prepends, trims to 1000, and re-serializes. This creates a performance cliff at cap since stdout/stderr are stored verbatim with no size limit — a single `journalctl` dump could push the blob past platform storage limits.

The integration hook fires audit writes as unawaited futures with no error boundary. A platform channel failure in `_storage.write()` becomes an unhandled async exception. The hook also instantiates a fresh `ExecutionAuditService()` on every invocation rather than reusing an instance.

The `AuditLogScreen` is fully implemented but unreachable — no route, drawer item, or navigation action references it anywhere in the app.

## Fire-and-forget audit with no safety net

The integration in `ai_copilot_sheet.dart` calls:

```dart
unawaited(ExecutionAuditService().logExecution(auditEntry));
```

There is no `.catchError` or zone error handler. The `_loadAll` method internally catches JSON parse failures and returns `[]`, so corrupted storage is contained at the read layer. But if `_storage.write()` itself throws (permissions, platform channel failure), the future completes with an error that becomes an unhandled exception in the current zone. On debug builds this crashes; on release builds the behavior depends on zone configuration.

The fix is either `.catchError((_) {})` on the future, or wrapping `logExecution` internals in a top-level try/catch that swallows storage-layer failures gracefully.

A new `ExecutionAuditService()` is instantiated each time `_runCommand` fires. The constructor creates a `FlutterSecureStorage` instance; while functionally correct (no mutable state), the service could be a `late final` field on `_AiCopilotSheetState` to avoid repeated allocation.

## Scalability of the JSON-blob storage pattern

The entire audit log lives under one FlutterSecureStorage key. At 1000 entries with full stdout/stderr, this string could easily reach hundreds of kilobytes. FlutterSecureStorage on Android uses EncryptedSharedPreferences with a practical size limit of ~1MB per value on some OEMs.

The `stdout` field is stored verbatim with no truncation. A command like `find / -type f` or `journalctl` produces megabytes of output. A single such entry would push the serialized blob over platform limits, causing silent write failures or crashes depending on the platform. Truncating stdout/stderr to a cap (4KB) before constructing the `ExecutionAuditEntry` in the hook would prevent this.

## Entry ID collision window

The ID is constructed as:

```dart
id: '${startTime.millisecondsSinceEpoch}_$index',
```

`startTime` is captured once before the command runs, and `index` is the plan-step index. If the same step is retried after a failure, `index` is identical and `startTime` could share the same millisecond. Duplicate IDs don't cause a storage error (both entries get stored), but `deleteEntry` uses `removeWhere((e) => e.id == id)` and would delete both entries — unintended data loss.

## Test coverage gaps

Not tested:

- The integration hook in `ai_copilot_sheet.dart` (no widget test verifying `logExecution` is called after command execution). A refactor of `_runCommand` could silently drop the audit hook.
- Error resilience of `_loadAll` when storage contains malformed JSON (the code handles it via try/catch returning `[]`, but no test exercises this path).
- The `AuditLogScreen` and `AuditLogEntryCard` widgets (no widget tests).
