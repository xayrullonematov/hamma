# Execution audit log for AI-proposed commands (v2)

The change adds an audit trail for every AI-copilot-proposed command execution: an immutable model, a FlutterSecureStorage-backed service with a 1000-entry cap, a Brutalist UI with search and filter chips, and integration into the copilot's `_runCommand` flow. All five issues from the v1 review have been addressed — error boundaries added, navigation wired, output truncated, service instance reused, and ID generation hardened with secure random suffixes.

**Watch for:** The `serverName` field is populated with the raw `serverId` rather than the human-readable name, so the UI displays opaque IDs in the header (confirmed). The AuditLogScreen loads all entries on every tab switch without caching or lifecycle gating, which will produce redundant secure-storage reads on tab-heavy navigation (likely).

**Verdict**: APPROVED

## High-level view

The v1 concerns are resolved cleanly. `.catchError((_) {})` silences unawaited audit failures. The `_auditService` field avoids per-call instantiation. Output truncation at 4096 chars with a `[truncated]` suffix prevents storage bloat. The ID now uses microsecond precision plus `Random.secure()` for collision resistance. Navigation wires the screen as the "Audit" tab in the sidebar's IndexedStack.

Two minor issues remain. The audit entry's `serverName` is set to `widget.serverId` (the opaque ID string, not the display name), so every entry card header shows an ID like `server_abc` instead of "Production". The `AiCopilotSheet` widget only receives `serverId`; passing a display name would require a minor API change.

The `AuditLogScreen` calls `_loadEntries()` unconditionally in `initState`, which fires every time the IndexedStack builds the tab. For a user switching tabs frequently, this means repeated deserialization of up to 1000 entries from encrypted storage on every switch — a latency spike on older devices, though not a correctness bug.

## serverName vs serverId mismatch

Both audit-entry construction sites in `_runCommand` set `serverName: widget.serverId`:

```dart
serverName: widget.serverId,
```

The `AuditLogEntryCard` renders this value as `entry.serverName.toUpperCase()` in the card header. Since `AiCopilotSheet` only receives a `serverId` field (no display name), the card will show the raw ID. This isn't a data-loss bug — the information is still correct — but it degrades the UI's readability. A follow-up could thread the server's `name` from `ServerProfile` into the widget, or look it up at log time.

## AuditLogScreen lifecycle and repeated loads

The screen creates a fresh `ExecutionAuditService` instance and calls `loadAll()` in `initState`. Within the dashboard's `IndexedStack`, Flutter may dispose and recreate inactive tabs depending on the `IndexedStack` behavior and keys. Even if the widget stays alive, a pull-to-refresh or manual "Refresh" button already exists. The main concern is the first load on tab activation causing a visible spinner while decrypting and parsing the full JSON blob — noticeable at high entry counts on lower-end Android hardware. This is not blocking but worth noting for a follow-up optimization pass.

## Test coverage status

The unit and integration tests cover model serialization, all service CRUD paths, ordering, trimming at cap, case-insensitive search, and per-server filtering. The v1 gap around widget tests for `AuditLogScreen` and the integration hook in `ai_copilot_sheet.dart` remains untested — neither is exercised via widget tests. This is acceptable for a first pass given the Flutter SDK is unavailable in this environment.
