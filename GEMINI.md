# Gemini Context: Hamma (AI Server V2)

This project, internally referred to as **Hamma** or **AI Server V2**, is a mobile-first Flutter application designed for managing remote servers directly via SSH from the device, augmented by AI-driven command suggestions and safety assessments.

## Project Vision & Core Principles

**Product One-Liner:** "Manage your server without writing commands."

1.  **Direct SSH First:** All SSH connections must happen **directly from the Flutter app** using `dartssh2`. No backend SSH proxies or session pooling should be used for core transport.
2.  **AI as Assistant, Not Transport:** AI is used to explain problems and suggest commands, but it does not replace the SSH layer.
3.  **Terminal as Fallback:** While raw terminal access (via `xterm`) is provided for power users, the main UX should prioritize simplified actions and buttons.
4.  **Safety Before Execution:** Never blindly execute AI-generated commands. Visibility, editability, and explicit user confirmation are mandatory.
5.  **Avoid "Fake States":** Do not report "Connected" or "Healthy" unless verified by an actual SSH check.

## Architecture

The project follows a feature-based architecture with a clear separation between core services and UI features.

- `lib/core/`: Foundation services and models.
    - `ai/`: AI command generation (`AiCommandService`) and risk assessment logic (`CommandRiskAssessor`).
    - `ssh/`: SSH connection management and command execution (`SshService`).
    - `storage/`: Local persistence for API keys (`ApiKeyStorage`) and server profiles (`SavedServersStorage`).
    - `models/`: Data structures like `ServerProfile`.
- `lib/features/`: Feature-specific UI and logic.
    - `ai_assistant/`: UI for interacting with the AI to generate commands.
    - `quick_actions/`: Pre-defined server management tasks (e.g., "Check logs", "Restart service").
    - `servers/`: Server profile management (Add/Edit/List/Test).
    - `terminal/`: Interactive SSH terminal screen.
    - `settings/`: App configuration (e.g., AI API keys).

## Tech Stack

- **Framework:** Flutter (Dart)
- **SSH Protocol:** `dartssh2`
- **Terminal Emulator:** `xterm`
- **Security:** `flutter_secure_storage` for credentials and saved hosts.
- **AI Integration:** OpenAI-compatible REST APIs.

## UX Principles

- **Simple First:** Non-experts should be able to restart services or inspect logs without terminal knowledge.
- **Buttons Over Commands:** Prefer high-level actions (e.g., "Install Docker") over raw shell commands.
- **Explain Everything:** Always explain what a command does and why it was suggested before execution.

## Command Execution Rules

### 1. AI-Generated Commands
- Must be visible and editable before execution.
- Require explicit user confirmation.
- Must include risk warnings generated by `CommandRiskAssessor`.

### 2. Dangerous Commands
Flag and require extra confirmation for operations involving:
- Deletion/Overwrite (`rm -rf`, `docker system prune`).
- Recursive permission/ownership changes (`chmod -R`, `chown -R`).
- System-wide changes (`reboot`, `shutdown`, `systemctl stop`).
- Firewall or SSH config changes (`iptables`, `ufw`).
- Database resets or destructive migrations.

## Building and Running

### Prerequisites
- Flutter SDK (matches version in `pubspec.yaml`).
- Android Studio / Xcode for mobile builds.

### Commands
- **Install Dependencies:** `flutter pub get`
- **Run the App:** `flutter run`
- **Run Tests:** `flutter test`
- **Build Android:** `flutter build apk`
- **Build iOS:** `flutter build ios`

## Development Conventions

- **Reliability & Simplicity:** Prioritize clean, maintainable code over overengineering.
- **Testing:** Add unit tests for core services and widget tests for UI features.
- **Risk Assessment:** Always use `CommandRiskAssessor` when presenting commands to users.
